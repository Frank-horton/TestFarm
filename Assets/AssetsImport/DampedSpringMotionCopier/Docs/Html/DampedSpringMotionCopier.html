Ever needed to attach one game object to another non-rigidly? In a soft fashion, sort of via a spring. This free script should help you.

The task of attaching one object to another in a Unity game via a spring is not uncommon. It's also not very hard. But most people do it in a straightforward fashion by simulating a real spring. Well, there is a better approach that has significant advantages over the simulation method. Namely, by using explicit formulas for a damped spring. A wikipedia article with the formulas is here.

The advantages of using the direct formulas are:

  • No need for a simulation.
  • The relative position of the driven object can be obtained at any time without intermediary steps. The computational cost is fixed and small.
  • The result is exact always, no simulation errors can creep in.
  • There are no stability issues, no matter how wild the input parameters used.

This approach is not widely used as it is tricky even to calculate positions with it, and much harder to calculate rotations. The factor-ordering problem for quaternions puts most programmers off. So people just use a simulation. But from now on you can just grab this free solution that just works and not worry about the maths.

Example uses:

  • Attaching a camera to a racing car with such a method may convey collisions more dramatically
  • Attaching the skinning key points to a moving rider with this method will create a perception of moving clothing
  • Rider can be attached to bike with this method.
  • Objects held by force-fields.

This free code is a part of a bigger animation/physics tool Physical Walk.

How to use

  1. Add a DampedSpringMotionCopier.cs script component to the object that you want to copy the motion onto.
  2. For positional and rotational parts specify the source objects to copy the motion from. These can be different - you can copy linear motion from one object and rotational motion from another. You can omit the source object for either the positional or rotational part altogether which means that the corresponding motion will not be changed.
  3. Adjust the parameters, most importantly damping criticality and natural frequency.

Script parameters

Source object - the object from which you want to copy the motion onto this object.

Motion copying type - whether you want to copy the motion instantly, via a spring, or leave it alone.

Damping criticality - the parameter that dictates how quickly the copied motion dies down. Values from 0 to 1 correspond to an under-damped spring, 1 to a critically damped spring, and anything above to an over-damped spring. See the Wikipedia article above for in-depth explanation.

An under-damped spring will oscillate. With a value of 0 it will oscillate forever without dying out. A value of 1 (critical damping) should copy the motion via the spring in the fastest possible fashion (for a given natural frequency). Some small oscillations can be expected if the source object is moving. With values over 1 the spring will be over-damped and the oscillations will be absent.

Natural frequency - this parameters tells how stiff the spring is. The higher the value the stiffer the spring will be and the quicker the target will match the motion of the source. The units of the parameter is inverse seconds. So if you want positions or orientations to match after roughly two seconds, try parameter values in the region of 0.5. Again, see the Wikipedia article for reference. For linear motion you can specify the value for all three axes independently.

Apply spring in moving reference frame - determines whether the spring is considered to be moving together with the source object or not. You likely want to have it checked, which is the default value. If the source object is moving with a constant speed the target object will eventually catch up only if the value is true.

Fix relative position at start - this allows you to specify the point towards which the spring pulls:

DONT_FIX_INITIAL_RELATIVE_POSITION - there will be no offset between the object centers.
IN_WORLD_FRAME - the spring will be trying to match the relative position of the target object that existed at the beginning (when Play Mode has started).
IN_SOURCE_LOCAL_FRAME - the spring will be trying to return the target object to its relative position with respect to the source object expressed in the source object's reference frame. This sentence is bound to have too many nouns :). It might be easier just to see the effect of the parameter when the source object is rotated in-game.

For a rotational spring only the rotational offset in the local reference frame of the source object can be fixed to a non-zero value.

Additionally, for a positional spring you can specify more advanced parameters such as the displacement scales and limits per each axis. The default values, which you will likely stick with, have no effect on the motion copying.

Notes on values

Setting criticality to anything higher than, say, 2 does not make much sense. You may think that high values will speed up settling down, but in reality this will lead to a longer matching time because the over-damping acts like a brake. If in addition to a high value of criticality you set a lower value of natural frequency, this will make the spring weak and over-damped, and it will not return the target object to the needed position at all.

Download directly 7z (extract to the folder Assets of your game), unitypackage or from Unity Asset Store.

All essential code is contained in the two files DampedSpringMotionCopier.cs and DampedSpringMotionCopier.Utils.cs, spanning one class. You can combine them into one cs file. The third file TooltipAttribute.cs is provided solely for tool-tips. You will be able to remove it too if a minimal interference with your project structure is necessary.
